❯ 为什么需要LLVM/libclang? 帮我解释下这里的技术栈依赖和原因                     

● 这个依赖链是这样的：

  依赖链

  whisper-rs (Rust wrapper)
    → whisper-rs-sys (FFI bindings)
      → whisper.cpp (C++ 语音识别引擎)
        → GGML/GGUF (模型推理框架)

  为什么需要 libclang

  whisper-rs-sys（版本 0.11.1）在编译时需要完成一个关键步骤：生成 Rust 到 C/C++ 
  的 FFI 绑定。

  它使用 bindgen 工具自动扫描 whisper.cpp 的 C 头文件（whisper.h），生成对应的
  Rust unsafe 函数声明。这个过程需要 libclang 来解析 C
  头文件中的类型、函数签名、结构体等。

  具体来说，bindgen 调用 libclang 的 C API 来：
  1. 解析 whisper.h 中的几千行 C 声明
  2. 将 C 类型映射到 Rust 类型（如 float* → *mut f32）
  3. 生成类型安全的 Rust 绑定代码

  为什么 macOS 不需要

  macOS 自带 Xcode Command Line Tools，其中包含 libclang.dylib，bindgen
  可以直接找到。Windows 没有内置的 LLVM/Clang，需要手动安装。

  解决方案

  你有两个选择：

  方案 A（推荐）：安装 LLVM
  winget install LLVM.LLVM
  # 然后设置环境变量
  $env:LIBCLANG_PATH = "C:\Program Files\LLVM\bin"

  方案 B：安装 Visual Studio Build Tools 2022，勾选 "C++ Clang tools for
  Windows" 组件。这会在 VS 安装目录下提供 libclang.dll。

  不影响我写的代码

  重要的是，这个编译错误与本次术语定制化功能完全无关。它是 whisper-rs-sys 的
  build script 阶段报错的，发生在编译器检查我的 Rust
  代码之前。项目本身之前也需要这个依赖才能编译。

  编译时 vs 运行时

  编译时（开发机）:                      运行时（客户机器）:
  ─────────────────                     ─────────────────
  Rust 源码                             编译好的 .exe / .dll
     ↓ cargo build                         ↓
  bindgen + libclang                     whisper.cpp 已经静态链接
     ↓ 生成 FFI 绑定                     进可执行文件里了
  whisper.cpp C++ 源码                      ↓
     ↓ C++ 编译器 (MSVC)                 直接调用 GGML 推理引擎
  whisper.cpp → .lib/.a                   运行 Whisper 模型
     ↓ 静态链接                           ✅ 不需要 LLVM
  最终 .exe (包含一切)                    ✅ 不需要 C++ 编译器
                                         ✅ 不需要 bindgen

  ├──────────────────────────────────────────────────────┼──────────────────┤
  │ 前端静态资源 (Next.js 导出)                          │ bindgen / 头文件 │
  ├──────────────────────────────────────────────────────┼──────────────────┤
  │ 预置术语 CSV 文件                                    │ 任何开发工具     │
  └──────────────────────────────────────────────────────┴──────────────────┘

  实际打包方式

  用 pnpm run tauri:build 构建时，Tauri 会生成一个 .msi 或 .exe
  安装包。whisper.cpp 的 C++ 代码会被 MSVC 编译成 .lib，然后静态链接到最终的
  meetily.exe 中。客户拿到的是一个自包含的 exe，双击就能跑。

  唯一的运行时依赖是 Whisper 模型文件本身（比如
  ggml-large-v3.bin），这个通常会放在安装包的 models/
  目录下，或者首次启动时自动下载。
