# PRD_TERMINOLOGY_CUSTOMIZATION_V3 审查意见

## 审查范围

- 审查对象：`PRD_TERMINOLOGY_CUSTOMIZATION_V3.md`
- 参考代码库：`meetily-forked`
- 审查方式：基于现有项目架构、已存在实现路径、数据模型、依赖约束和交付复杂度进行静态审查
- 审查目标：识别该 PRD 在产品设计、技术实现、审计合规、项目管理和成本控制上的主要风险，并给出修订建议

## 总体结论

这份 V3 版 PRD 相比前一版有明显进步，尤其是在以下方面：

- 增加了 `Phase 0` 基线测量，说明开始重视“先测量、后承诺”的基本方法论
- 将 `L3` 明确为默认仅生成建议，降低了大模型直接覆盖正文的风险
- 补充了缓存一致性、幂等迁移、MVP 裁剪等工程化考虑
- 对阶段目标和功能边界做了一定收敛，方向比上一版更接近可执行

但从“是否可以据此直接立项并进入开发”的标准看，这份 PRD 仍然不够成熟。

当前版本仍存在三类核心问题：

1. 存在至少一个方案级硬伤，导致文档中的关键实现方式在当前技术栈下不能直接落地
2. 审计与证据链设计仍不完整，不满足高风险专业场景对原始转录结果保全和可追溯性的要求
3. 部分看似合理的交互与流程设计，缺少与现有项目真实架构的对齐，实际落地成本被低估

因此，我的结论是：

- `V3` 可以作为方向性方案继续迭代
- `V3` 还不适合作为直接排期开发的正式 PRD
- 建议在进入编码前，至少再完成一轮 `V3.1` 修订

## 关键结论摘要

### 可以肯定的部分

- 业务场景明确，目标用户和价值主张清楚
- 功能分层思路总体正确，`L1/L2/L3` 的抽象有助于后续做灰度和裁剪
- 对风险和 MVP 的意识明显增强
- 已开始考虑缓存、迁移、建议流、用户确认等工程问题

### 目前仍不过线的部分

- `L2` 的核心正则边界实现与当前 Rust 依赖不兼容
- 原始 STT 输出没有被强制保留，证据链不完整
- `L3` 的“逐条接受/拒绝”设计没有充分的数据模型支撑
- 文档叙述的后处理链路与当前项目真实保存链路存在偏差
- 性能、自动降级和可配置性方面仍然存在过度承诺

## 严重问题

### P0-1：L2 的正则边界方案在当前技术栈下不能直接实现

#### 结论

PRD 中 `L2` 的日语/中文 whole-word 边界方案，仍然依赖 Rust 标准 `regex` 不支持的 look-around 语法。

这意味着：

- 文档中的关键示例不能按现有依赖直接编译
- 当前实现路径不是“效果待验证”，而是“方案本身不成立”
- 如果不先修改技术方案，研发会在实现阶段才暴露根本性阻塞

#### 问题本质

PRD 中使用了类似以下思路：

- `(?<!...)`
- `(?!...)`

这类写法在很多正则实现里可用，但当前项目使用的是 Rust 标准 `regex` 库。该库出于性能和安全原因，不支持 look-around。

因此，文档把“不可直接实现的伪代码”写成“计划中的标准实现”，会误导开发和排期。

#### 影响

- `L2` 这一层无法按 PRD 原样落地
- “中文/日语不误替换”的核心承诺没有实现基础
- 性能估算、缓存设计、排序逻辑等后续优化都建立在一个尚未成立的底座上

#### 建议

必须在 `V3.1` 中二选一：

1. 明确改用其他能力更强但成本更高的实现路线
2. 放弃当前 whole-word 正则边界承诺，先收敛到可实现的窄规则集

更现实的建议是：

- MVP 阶段只支持严格精确匹配和显式变体表
- 不在第一版承诺复杂的中日文自然边界识别
- 如果必须支持 whole-word，应单独写一个技术预研章节，而不是直接写进主方案

### P0-2：原始 STT 输出没有被定义为必须保留，审计链不完整

#### 结论

当前 PRD 默认将 `L1+L2` 后的文本作为主存储结果，而原始 STT 输出是否保留，被写成了“如有需要再加字段”。

对于一般办公软件，这种设计也许只是欠稳妥；但对于专业术语、危化品、安全合规等场景，这属于严重设计缺口。

#### 问题本质

如果系统先做术语替换、纠错或规范化，再把“处理后的结果”当正文持久化，那么后续就很难回答以下关键问题：

- 原始模型当时到底识别出了什么
- 哪些内容是模型原文，哪些内容是规则替换的
- 哪些内容是 LLM 建议的
- 用户最终确认时看到的是哪个版本
- 导出、归档、审计时引用的是哪个版本

没有原始 STT 文本作为第一证据源，后续所有“审计”“追责”“回溯”都会失去锚点。

#### 影响

- 不能满足严格场景下的可追溯要求
- 出现误修正时，难以可靠复盘
- L2 和 L3 同时存在时，责任归因会变得模糊
- 后续如果要补做审计，数据迁移成本会显著升高

#### 建议

将以下原则提升为 `Phase 1A` 的硬要求：

- 原始 STT 输出必须单独保存
- `L1/L2` 处理结果与原始文本必须明确区分
- `L3` 建议与最终确认版本必须分层存储

建议最少保留四层：

- `raw_transcript`：原始 STT 输出
- `normalized_transcript`：L1/L2 后结果
- `l3_suggestions`：L3 候选建议
- `final_transcript`：用户确认后的最终版本

如果不想一次性全部实现，至少也应先确保：

- `raw_transcript`
- `current_transcript`
- `correction_history`

这三类信息是完整可回放的。

## 高优先级问题

### P1-1：L3 的逐条接受/拒绝设计，当前数据结构仍不足以支撑

#### 结论

PRD 希望用户可以逐条查看建议、逐条接受、逐条拒绝、批量处理并保留历史，但当前设计的数据粒度仍然偏粗。

#### 问题本质

如果只保存：

- 原文整段
- 改后整段
- 状态

那么很多交互看起来能画出来，但不代表系统真的能可靠支撑：

- 两条建议落在同一段文本时如何处理冲突
- 先接受第 2 条、再拒绝第 1 条时如何回放
- 批量接受后如何撤销部分修改
- 当正文后续被用户手动编辑时，旧建议是否失效

这些问题本质上都要求系统持有更细粒度的“建议补丁”信息，而不仅是整段 before/after 文本。

#### 建议

建议把 `L3` 从“整段替换”转成“建议补丁模型”，至少补齐：

- 建议作用范围或字符位置
- 建议来源
- 建议生成时所对应的文本版本
- 应用顺序
- 冲突状态
- 是否已过期
- 接受/拒绝人和时间

否则 UI 虽然能做出“建议列表”，但底层并不稳定。

### P1-2：L3 的目标定义仍不稳定，纠错与扩写被混在一起

#### 结论

PRD 文本中一方面要求 `L3` 不得添加原文没有的信息，另一方面示例中又展示了术语扩写、括号补充、解释性增强。

这是一个原则层面的冲突，不是文案细节问题。

#### 为什么严重

“术语纠错”和“术语扩写/知识补全”是两个完全不同的产品目标：

- 术语纠错：目标是尽量接近原始发言，只修正明显识别错误
- 术语扩写：目标是提高可读性或专业性，可能引入原文未明确说出的补充信息

一旦把两者混在同一个模式里：

- 验收标准会混乱
- 审计责任边界会模糊
- 用户将难以理解“系统到底改了什么”
- 专业高风险场景下会带来合规争议

#### 建议

必须在 PRD 中明确二选一，或者拆成两个独立模式：

1. `纠错模式`
2. `扩写模式`

并分别定义：

- 是否允许添加解释
- 是否允许补全简称全称
- 是否允许括号说明
- 是否允许把通用词替换为更专业术语

如果产品定位是“合规优先”，建议第一版只保留纠错模式。

### P1-3：文档中的后处理流程与当前项目真实架构仍未完全对齐

#### 结论

PRD 叙述中的保存与后处理流程，默认更多职责在 Rust 端串行完成；但当前项目实际上是由前端监听停止事件，再发起保存动作。

这意味着文档并没有完全贴合当前项目的事实架构。

#### 为什么会出问题

如果不先统一“谁负责聚合文本、谁负责保存、谁负责发起 L3、谁负责刷新页面状态”，后续非常容易出现：

- Rust 和前端都尝试触发后处理
- 状态更新顺序不一致
- 页面早于异步修正完成就展示旧数据
- 保存逻辑和纠错逻辑互相耦合

#### 建议

PRD 应单独补一节“与现有录音停止链路的集成方式”，明确：

- 录音停止事件由谁发布
- transcript 聚合由谁负责
- L1/L2 在实时流阶段还是保存阶段生效
- L3 由谁异步触发
- 页面如何感知建议已生成
- 摘要生成默认依赖哪一层文本

在没有把这条链路重新画清楚之前，不建议进入开发。

### P1-4：性能承诺仍偏乐观，缺少与真实数据规模匹配的基准

#### 结论

PRD 在性能上已有收敛，但对 `L2` 热路径和 `L3` 处理开销的表述仍偏乐观，尤其是当术语表规模、语言混合程度和会议长度上升后。

#### 风险点

- 每个实时 chunk 都要做规则匹配和字符串替换
- 若术语表扩展到数百条，遍历成本和内存分配成本都会上升
- 当会议较长、转录段较多时，后处理的总成本可能远高于预期
- 若未来再加入建议去重、diff、缓存失效控制，实际耗时会进一步增加

#### 建议

把所有性能承诺改成“以 Phase 0 测量结果为准”，并补一张最少的测试矩阵：

- 术语数量：50 / 200 / 500
- 语言模式：日文 / 中文 / 英文 / 混合
- chunk 长度：短 / 中 / 长
- 设备等级：低配 / 中配 / 高配

没有真实测量数据前，不要把 `<1ms` 一类指标写成既定结论。

## 中优先级问题

### P2-1：L2 不允许单独关闭的设计不稳妥

#### 结论

PRD 认为 `L2` 是确定性规则替换，因此不需要单独关闭。这个判断不成立。

#### 原因

确定性只代表：

- 结果稳定
- 可复现

不代表：

- 一定正确
- 一定无害
- 一定适合所有会议、所有用户、所有术语包

在专业场景里，稳定地替换错比偶发错误更危险，因为它会批量污染结果。

#### 建议

至少支持以下一种或多种关闭粒度：

- 全局关闭 `L2`
- 按术语包关闭
- 按会议类型关闭
- 按语言关闭

最低限度也要支持调试与排障时的临时关闭能力。

### P2-2：自动硬件/电源降级策略的实现前提被高估

#### 结论

PRD 中关于设备能力识别和电源状态驱动降级的设计目标合理，但当前项目基础能力并不足以支撑如此自动化的调度。

#### 风险

- 现有硬件检测较粗
- 设备能力判断可能不稳定
- 电池状态在当前代码路径中并无成熟基础
- Windows、macOS、Linux 跨平台行为很可能不一致

#### 建议

把这部分从“自动决策逻辑”降为：

- 推荐策略
- 可选增强项
- 后续阶段能力

第一版尽量采用显式配置和人工可控策略，而不是依赖自动判断。

### P2-3：CSV 与编码兼容能力被低估

#### 结论

PRD 已考虑到 UTF-8 BOM、Shift-JIS、预览和预置词库导入，这个方向是正确的，但工作量与边界复杂度被低估了。

#### 风险

- 编码自动识别不是简单功能
- CSV 中引号、逗号、换行、表头异常都可能出错
- 术语去重、归一化、冲突检测会放大复杂度
- 不同客户的词库质量差异很大

#### 建议

建议拆成独立工作包：

- MVP 先只支持 UTF-8 BOM
- Shift-JIS 作为增强项，不要在第一版承诺“自动检测可靠”
- 导入时必须有预览、错误行提示和去重策略

## 项目管理与排期意见

### 当前问题

当前 PRD 的阶段拆分已比上一版更合理，但整体上仍偏向“编码视角排期”，而不是“从验证到上线的真实交付排期”。

被低估的工作通常包括：

- 方案预研
- 架构设计评审
- 数据迁移设计
- 回归测试
- 真实词库清洗
- UI 交互打磨
- 试运行反馈和二次调整

### 建议的工作包拆分

建议不要把所有内容作为一个完整功能包直接推进，而是拆成以下四个工作包：

1. `工作包 A：L1/L2 技术预研与基线测量`
2. `工作包 B：原始文本与审计数据模型`
3. `工作包 C：L3 建议流与人工确认机制`
4. `工作包 D：词库导入、预置包与运营治理`

这样拆分的好处是：

- 每一块都可独立验收
- 技术风险不会后置到编码阶段才暴露
- 可先做高价值低风险能力，再逐步上更复杂能力

### 建议的实施顺序

建议顺序如下：

1. 先确认 `raw_transcript`、版本分层和建议模型
2. 再完成 `L1/L2` 的最小可用版
3. 然后做 `L3` 建议流，而不是自动覆盖正文
4. 最后再做词库导入、自动降级和行业预置包

### 对成本的判断

如果以“能演示”为目标，现有排期或许勉强可压缩。

但如果以“能稳定上线并承担专业场景责任”为目标，当前成本估算偏低。

真正的成本不在于多写几个模块，而在于：

- 降低误修正风险
- 保证可追溯
- 管住复杂配置
- 做出用户可以信任的建议流

## 产品层建议

### 建议收紧第一版目标

第一版不要试图同时解决：

- 提升识别准确率
- 自动纠错
- 自动扩写
- 行业模板导入
- 多级性能自适应
- 完整审计闭环

这些目标叠加后，复杂度会急剧上升。

更现实的第一版目标应是：

- 保留原始转录
- 支持少量高价值术语的可控纠错
- 对大模型建议只做“建议”，不做自动覆盖
- 提供最小可用的审计记录

### 建议重写成功指标

不要只写“关键术语准确率 > 99%”这类单一目标。

建议拆成多个指标：

- `L1` 是否提升了目标术语的初始识别率
- `L2` 的误替换率是多少
- `L3` 建议接受率是多少
- 用户人工复核时长是否下降
- 高风险术语误修正率是否低于阈值

这样更有利于做灰度和复盘。

## 建议的 V3.1 修订方向

### 必改项

1. 重写 `L2` 边界实现章节，去掉当前不可直接落地的标准实现描述
2. 将 `raw_transcript` 提升为必须项，而不是可选增强
3. 重新定义 `L3` 的建议数据结构，明确支持逐条建议而不是整段替换
4. 明确“纠错模式”和“扩写模式”是否拆分
5. 补一节“与当前录音停止/保存链路的集成方案”

### 建议改项

1. 降低自动硬件/电源调度的承诺级别
2. 允许 `L2` 在调试或特定场景下关闭
3. 把 CSV 和编码支持单独拆为后续工作包
4. 把摘要系统和术语系统之间的依赖关系写清楚

## 最终结论

这份 `V3` 不是一份失败的 PRD，相反，它已经比上一版更接近一个“可工程化讨论”的方案。

但它仍然不是一份可以直接交给研发进入实现的正式文档。

当前最关键的判断是：

- 问题已经不再主要是“想法对不对”
- 而是“方案是否真的能在当前架构和依赖约束下成立”

在这个标准下，`V3` 仍未完全过线。

我的建议是：

- 不要急于进入编码
- 先产出一个 `V3.1`
- 把 `L2` 可实现性、原始证据链和 `L3` 建议模型三件事彻底定下来

只要这三件事定清楚，后续开发风险会下降非常多；反之，如果现在直接开工，后面大概率会在实现中途返工。
