# PRD_TERMINOLOGY_CUSTOMIZATION_V3.1 审查意见

## 审查范围

- 审查对象：`PRD_TERMINOLOGY_CUSTOMIZATION_V3.1.md`
- 参考代码库：`meetily-forked`
- 审查方式：基于当前项目实际架构、已有数据流、Rust/TypeScript 技术约束、现有保存与摘要链路进行静态审查
- 审查目标：确认 V3.1 是否已经修复上一版的关键硬伤，并识别剩余的实现级风险、审计缺口和项目管理问题

## 总体结论

`V3.1` 相比 `V3` 已有明显进步，而且是“实质性进步”而不是“文案层优化”。

本轮最值得肯定的修正包括：

- 明确将 `raw_transcript` 提升为强制保留的一等数据，补上了审计链的核心锚点
- 承认 Rust `regex` 不支持 look-around，并改为使用捕获组模拟边界
- 将 `L3` 从整段替换模型升级为“逐条建议补丁 + 版本冲突检测”
- 将 `L3` 第一版明确限定为“仅纠错，不扩写”
- 显式对齐当前项目中“Rust 发布事件、前端聚合、再保存”的真实录音停止链路
- 对自动降级、CSV 编码支持和成功指标的表述明显更加克制

因此，和上一轮相比，这份文档的状态已经从：

- “仍不适合立项”

提升到了：

- “可以进入详细技术设计阶段，但仍不适合直接按文档开始完整编码”

原因在于，`V3.1` 虽然已经修复了大部分方向性错误，但还保留了几类“实现级精度问题”。这些问题不是战略方向错误，但一旦不在 PRD 阶段先纠正，研发在实施时仍然大概率返工。

## 本轮结论摘要

### 已明显修复的问题

- `L2` 不再依赖 Rust 不支持的 look-around
- `raw_transcript` 已从“可选增强”升级为“强制保留”
- `L3` 不再默认覆盖正文，而是建议流
- `L3` 已开始考虑版本冲突和并发控制
- 与现有录音停止链路的职责划分已经比上一版清晰得多

### 仍然不过线的部分

- `L2` 示例代码本身仍有实现级 bug
- `raw vs normalized` 双轨数据流尚未和当前事件模型真正闭环
- 审计链仍缺少“规则版本/术语快照版本”这一层证据
- “按术语包禁用/治理”的能力没有在表结构中真正落地
- `L3` offset 语义未完全定义，前后端 Unicode 处理存在潜在错位风险

## 严重问题

### P0-1：L2 示例实现仍然存在可直接导致错误行为的代码级问题

#### 结论

`V3.1` 虽然方向上修正了 look-around 方案，但文档中给出的替代实现仍然存在两个会直接影响结果的细节错误。

#### 问题 1：日语边界字符类拼接方式有误

文档中先定义：

```rust
let boundary = r"[\p{Han}\p{Hiragana}\p{Katakana}ー]";
```

随后又在 pattern 中使用：

```rust
format!(r"(^|[^{b}]){escaped}($|[^{b}])", b = boundary, escaped = escaped)
```

这会造成边界字符类被二次嵌套，拼接结果不是期望的表达式。

这类问题不是“性能优化不足”，而是“按文档抄代码会写出错误 regex”。

#### 问题 2：中文 whole-word 规则的捕获组数量与替换模板不一致

中文分支当前生成的是：

```rust
(^|[^\p{Han}])({term})([^\p{Han}]|$)
```

这实际上有三个捕获组：

1. 前边界
2. 术语自身
3. 后边界

但统一替换模板仍写成：

```rust
${1}replacement${2}
```

这会把原术语分组错误地当成尾边界使用，替换逻辑是错的。

#### 影响

- `L2` 虽然概念上可行，但当前示例代码不能直接作为实现依据
- 中文规则最容易在开发阶段出现“看起来编译通过，但结果不对”的隐蔽 bug
- 文档作为 PRD/技术依据的可信度会被削弱

#### 建议

必须在 `V3.1.1` 中修正 `7.3` 的示例实现，至少做到：

- 日语边界字符类的构造方式语法正确
- 中文规则与替换模板严格配对
- 不同语言的替换模板不要强行共用一套占位假设
- 文档中的示例代码必须能够被研发直接翻译为可运行实现

## 高优先级问题

### P1-1：raw / normalized 双轨模型仍未和当前事件数据流真正闭环

#### 结论

`V3.1` 已正确识别出“必须同时保存 raw_transcript 和 normalized_transcript”，但文档仍然低估了它对当前事件结构、前端状态和保存接口的改造成本。

#### 当前项目现实

当前转录事件 `TranscriptUpdate` 只有一个 `text` 字段，事件发出时并没有同时携带：

- 原始 STT 文本
- L2 校正后文本

当前前端保存链路也只接收一个 `transcripts` 数组，并调用既有的 `api_save_transcript`。

也就是说，当前系统里并不存在天然的“双轨文本版本”。

#### 问题本质

PRD 假设：

- 前端能同时拿到每个 chunk 的 raw 和 normalized
- 前端能分别聚合并在停止录音时同时提交

但这在现有实现中并不是“加一个字段就结束”，而是至少涉及：

- Rust `transcript-update` 事件载荷扩展
- 前端 transcript 状态结构扩展
- 保存接口改造
- 与现有 meeting details / pagination / summary 读取逻辑的兼容

#### 建议

在 PRD 中明确增加一个“小节级别”的接口改造说明：

1. `TranscriptUpdate` 事件是否扩展为：
   - `raw_text`
   - `normalized_text`
   - `display_text`
2. 前端聚合时以哪个字段为显示来源
3. 保存时是按段保存双版本，还是只在会议级别保存双版本
4. 现有 `storageService.saveMeeting()` 如何迁移到新命令

只有这一步写清楚，`raw_transcript` 才不是纸面设计。

### P1-2：审计链仍缺少“规则版本”与“术语快照版本”

#### 结论

`V3.1` 已经保留了原始文本，这是非常大的改进；但“为什么会产生这个规范化结果”这件事，仍然缺少足够证据。

#### 当前设计的问题

文档认为：

- `raw_transcript` 可保留原始输出
- `normalized_transcript` 可保留 L1+L2 后结果
- L2 由于是确定性替换，因此可以通过对比重放

问题在于：术语表本身是会变化的。

如果后续发生以下情况：

- 某条术语被编辑
- 某个 CSV 导入覆盖旧规则
- 某个预置包升级了内容

那么未来即便你拿到了 raw 和 normalized，也未必能可靠还原“当时到底用了哪一套规则”。

#### 建议

建议至少补充以下一种或多种机制：

- `terminology_snapshot_hash`
- `ruleset_version`
- `import_batch_id`
- `rule_id` 级别的替换记录

如果第一版不想细到“逐次 L2 替换记录”，至少也要保留“本次转录所对应的术语快照版本”。

### P1-3：按术语包治理的产品能力还没有真正落到数据模型

#### 结论

PRD 中已经提出：

- 支持按术语包暂停/启用
- 支持预置术语包
- 支持词库治理

但表结构仍然没有提供足够的“术语归属”信息。

#### 问题本质

当前 `terminology` 表只有：

- `category`
- `priority`
- `language`

这不足以表达：

- 这条词来自哪个预置包
- 这条词来自哪次 CSV 导入
- 这条词是系统内置还是用户自定义
- 是否允许包级停用、升级、回滚

如果没有这类字段，“按术语包禁用”最后很可能只能在 UI 层模拟，而无法成为可靠的数据能力。

#### 建议

建议在 `terminology` 表中补充其中一些字段：

- `source_type`：`preset | imported | manual`
- `package_id`
- `package_name`
- `import_batch_id`
- `is_builtin`

这样后续：

- 包级启停
- 预置包升级
- 导入批次回滚
- 审计来源查询

才有真正落点。

### P1-4：L3 offset 语义仍需进一步精确定义

#### 结论

`V3.1` 的 `L3CorrectionSuggestion` 已经比 `V3` 完整很多，但 `char_offset` 的语义定义仍不够严格。

#### 风险来源

跨 Rust 与 TypeScript 时，最常见的问题不是“没有 offset”，而是：

- Rust 侧按字节索引
- JS/TS 侧按 UTF-16 code unit
- 人类理解按“字符”

在日语、中文、混合标点和特殊字符场景中，这三套语义不一致会导致：

- 前端高亮位置不对
- 接受建议时替换错位
- 版本冲突检测误判

#### 建议

PRD 应明确：

- `char_offset` 的计量单位到底是什么
- 是否需要同时保存 `end_offset`
- 是否只允许后端基于原文重新应用 patch，而前端仅用于展示
- 前后端是否统一采用“Unicode scalar value 索引”或其他明确规则

如果这一点不写清楚，后续最容易在多语言文本里出隐蔽 bug。

## 中优先级问题

### P2-1：L1 prompt 快照写入普通日志，可能带来信息泄露风险

#### 结论

PRD 当前将“L1 注入了什么”与“日志中的 prompt 快照”绑定，这在调试上方便，但在客户术语可能包含敏感信息时不够稳妥。

#### 风险

- 术语表本身可能是客户专有知识
- 运行日志通常不具备与业务数据相同的访问控制
- 调试日志更容易被导出、上传或长期保留

#### 建议

建议把两类能力分开：

- 运行日志：只记录“注入了多少条”“是否发生截断”
- 审计快照：单独存储受控元数据，不进入常规日志

第一版如果不做审计快照表，也至少不要默认把完整 prompt 内容写入普通日志。

### P2-2：L3 自动接受虽然默认关闭，但对于目标行业仍然偏激进

#### 结论

`V3.1` 已经明显收敛，默认值也改成了关闭，这个方向是对的。

但如果当前目标行业确实是危险化学品制造业，首版 PRD 中仍保留 `llm_correction_auto_accept` 配置项，会带来额外治理成本。

#### 原因

“存在配置项”通常意味着：

- 后续有人会要求把它打开
- 研发需要为它承担实现和测试成本
- 产品要对错误自动接受的后果有说明与责任边界

#### 建议

如果第一版目标强调“纠错可追溯、合规优先”，建议：

- 在 V1 范围内移除自动接受
- 或至少明确写成“仅保留字段，不在第一版 UI 暴露”

这样比“默认关闭但可打开”更符合场景风险等级。

## 项目管理与实施判断

### 本轮判断

和上一版相比，`V3.1` 已经不是“方向没想明白”的问题了。

现在主要问题变成：

- 文档离“可直接编码”还差最后一层实现精度

换句话说：

- 可以进入详细技术设计
- 可以开始接口级别拆分
- 可以开始预研和最小 POC
- 但不建议直接进入全量功能开发

### 为什么还不建议直接开工

因为当前剩余问题的特征是：

- 看起来不大
- 但都属于“做错一次就要返工很多文件”的类型

尤其是：

- `TranscriptUpdate` 事件模型
- 前端 transcript 状态结构
- 数据库存储方式
- L3 patch 应用规则

这些一旦先写错，后续会影响多个层次。

### 建议的推进方式

推荐在正式开发前补一个简短的技术设计稿或 RFC，专门回答以下问题：

1. `transcript-update` 事件是否扩展为双文本版本
2. 会议保存时的 raw / normalized 是按段持久化还是按整段持久化
3. `L2` 审计证据的最小可行版本是什么
4. `L3` offset 的统一语义是什么
5. 术语包和导入批次如何建模

只有这些问题定清楚，研发才能真正稳定推进。

## 建议的 V3.1.1 修订项

### 必改项

1. 修正 `7.3` 中日语边界字符类构造和中文替换模板的示例代码
2. 在 `9`、`11`、`12` 章补充 `TranscriptUpdate` 双轨载荷设计
3. 在 `10` 章补充术语来源/术语包/导入批次相关字段
4. 在 `4` 章增加术语快照版本或规则版本的审计信息
5. 在 `8` 章明确 `char_offset` 的精确定义和前后端一致性规则

### 建议改项

1. 将 L1 prompt 完整快照从普通日志中移出
2. 将 `llm_correction_auto_accept` 降级为后续版本能力，或至少不在 V1 UI 暴露
3. 明确摘要系统与 normalized_transcript 的长期关系，以及接受 L3 后是否自动提示重新摘要

## 最终结论

`V3.1` 是一次成功的修订。

它已经修复了上一版最致命的几个问题，说明文档作者已经开始从“想法驱动”转向“受架构和实现约束的工程设计”。

但从交付标准上看，`V3.1` 仍然还差半步：

- 它已经足够好，可以进入技术设计
- 但还不够精确，不适合作为研发直接照着实现的最终 PRD

如果继续推进，我的建议是：

- 不要重写整份文档
- 直接产出一个 `V3.1.1`
- 只修本轮指出的这几个实现级问题

这样成本最低，收益最高，也最有利于尽快进入稳定开发阶段。
