# PRD_TERMINOLOGY_CUSTOMIZATION_V3.2 审查意见

## 审查范围

- 审查对象：`PRD_TERMINOLOGY_CUSTOMIZATION_V3.2.md`
- 参考代码库：`meetily-forked`
- 审查方式：结合当前代码架构、事件流、数据模型、前后端类型系统与摘要链路进行静态审查
- 审查目标：确认 `V3.2` 是否已经修复 `V3.1` 剩余问题，并识别仍可能在实现阶段引发返工的风险点

## 总体结论

`V3.2` 是目前为止质量最好的一版，也是第一版真正进入“可以开展详细技术设计和 POC”的 PRD。

与 `V3.1` 相比，这一版有以下实质性改进：

- 补上了 `TranscriptUpdate` 双轨扩展设计
- 将术语快照哈希纳入审计链
- 将术语来源、术语包治理、导入批次追踪正式写入数据模型
- 对 `auto_accept` 做了更稳妥的收敛
- 对 offset 语义、L3 恢复机制、前端双轨聚合等问题进行了明确化

因此，这份文档已经明显优于前几版，不再是“方向有问题”，而是进入了“实现细节需再抛光”的阶段。

但它仍然不是完全可以让研发“照着写就行”的最终文档。

当前剩余问题集中在：

1. 个别方案前提仍然缺少验证，却被写成了已定实现
2. 某些简化设计会引入新的语义风险
3. 对当前项目真实代码路径的对齐还差最后一层精确性

因此我的结论是：

- `V3.2` 已具备进入技术详细设计和最小 POC 的条件
- `V3.2` 仍建议先做一个小修版（例如 `V3.2.1`），再作为正式开发依据

## 本轮结论摘要

### 已明显修复的问题

- 双轨转录事件模型已经被正面设计，而不是停留在抽象层
- 审计链从“只有 raw”升级到“raw + snapshot hash”
- 术语包治理能力开始真正进入表结构
- `auto_accept` 被降为保留字段但不在 V1 UI 暴露
- 前端双轨聚合、L3 恢复、摘要重新生成提示等交互关系被补充

### 仍需警惕的问题

- `L1` 的 token 计数方案可能依赖错误 tokenizer 假设
- `L3` 的“全文所有匹配统一接受”存在上下文误用风险
- `L3` 队列伪代码尚未充分表达真正的持久化排队语义
- 当前项目实际监听与保存链路，文档仍未完全精确对齐到代码结构
- `snapshot hash` 还不是完整可回放的规则证据

## 严重问题

### P0-1：L1 精确 token 计数方案的前提仍未被可靠证明

#### 结论

文档在 `V3.2` 中将 `L1` token 计数升级为“精确计数”，这是一个好方向，但当前实现假设本身仍然需要严格验证。

文档中直接写了：

```rust
use tiktoken_rs::cl100k_base; // Whisper 使用 cl100k_base 词表
```

这类描述的风险在于：一旦 tokenizer 假设与 Whisper 实际使用方式不完全一致，那么所谓“精确计数”其实只是“更复杂但未验证的估算”。

#### 为什么这很重要

`L1` 的核心约束是 224 token 左右的 prompt 长度限制。

如果 token 计数本身不准确，就会出现两类后果：

- 明明还能放下的术语被过早裁掉，降低 L1 覆盖率
- 实际超限却未被发现，运行时才报错或 silently degrade

这会直接影响：

- high priority 术语保留率
- 前端“已包含 / 未包含”提示的可信度
- Phase 1B 的验收准确性

#### 建议

建议将这部分从“既定实现”降级为“Phase 0 强验证项”，PRD 应明确：

- `tiktoken-rs` 只是候选方案
- 具体 tokenizer 是否与 Whisper 当前实现一致，必须先验证
- 若无法确认，则此方案回退为“保守估算 + 运行时截断容错”

在验证完成前，不建议把 `cl100k_base` 写成确定事实。

## 高优先级问题

### P1-1：L3 的“查找替换规则 + 全文 N 处”设计，存在上下文过度应用风险

#### 结论

`V3.2` 为了降低 offset 复杂度，把 LLM 产出从“逐 occurrence patch”改成了“查找词 + 替换词 + 后端全文搜索所有出现位置”。

这个设计比之前更容易实现，也更适合聚类 UI，但它并不天然安全。

#### 风险本质

LLM 在生成建议时，往往是基于某个具体上下文判断一段文字有误。

如果后端将：

- 一次上下文中的判断

扩展成：

- 全文所有相同字符串统一处理

就可能出现以下问题：

- 同一错误串在不同语境中，只有部分应该修正
- 某一处是术语误识别，另一处其实是用户有意表达
- 批量接受会把“局部纠错”放大成“全局规则”

这在危化品、法规术语和代码编号场景里尤其需要谨慎。

#### 建议

建议在 PRD 中明确把这件事写成“受控简化”，而不是默认安全假设：

- 第一版允许“全文 N 处统一处理”
- 但必须在 UI 明确提示“当前建议基于全文同串匹配”
- 对高风险术语应支持展开逐条上下文确认
- 后续版本预留回到 occurrence 级 patch 的可能

如果不写清楚，产品和研发很容易把它理解成“天然正确的批量纠错”。

### P1-2：L3 队列伪代码仍未完全表达“真正排队”的实现语义

#### 结论

文档已经意识到并发 OOM 风险，也引入了 `Semaphore(1)`，这是正确方向。

但目前展示的伪代码中，`try_acquire()` 失败后直接返回 `Ok(())`，并用注释说明“实际等待逻辑在 spawn 的 task 中”。这意味着文档本身还没有把“任务怎么排进去、怎么恢复、怎么确认没丢”写完整。

#### 问题本质

研发在按文档实现时最容易犯的错误是：

- 返回了 `queued`
- 但后端并没有真正持久化待执行任务
- 或者任务排进了内存，但 App 关闭就丢了

如果这层语义不在 PRD 里说明清楚，后续实现极容易出现“看起来有排队状态，实际上没有可靠队列”的半成品。

#### 建议

建议 PRD 在 `L3` 队列章节补齐以下信息：

1. 任务入队时是否立即写 DB 状态
2. `queued` / `running` / `done` / `failed` 的状态迁移表
3. App 重启恢复依据的字段是什么
4. 超时后是保留失败记录还是清空
5. 同一 meeting 是否允许重复提交 L3 任务

只要把这几件事写清楚，队列设计就会从“示意图”变成“可实现规范”。

### P1-3：架构对齐程度已明显改善，但当前链路描述仍不够精确

#### 结论

`V3.2` 已经比之前更贴近项目真实结构，但在“当前链路”章节里，仍然把部分职责落点写得过于页面化。

#### 当前代码现实

基于代码库，当前行为并不是简单的：

- `page.tsx` 监听 → 聚合 → 保存

更准确地说是：

- `transcript-update` 的主监听和聚合发生在 `TranscriptContext`
- 停止录音后的保存动作发生在 `useRecordingStop`
- 实际 IPC 封装在 `storageService`

这说明 PRD 的方向已对，但责任归属仍应写得更接近真实模块边界。

#### 为什么重要

如果 PRD 在这一层仍然不够精确，后续开发拆任务时就会出现偏差：

- 有人以为改页面组件即可
- 有人漏掉 `TranscriptContext` 的双轨状态改造
- 有人只改保存命令，忘了 IndexedDB / reload sync / transcript history

#### 建议

建议将文档中的“前端 page.tsx”表述统一修正为更准确的模块链路：

- `TranscriptContext`
- `useRecordingStop`
- `storageService`
- `recordingService`

这种修改不需要改设计，只需要把实现落点说准。

### P1-4：save_transcript_with_terminology 的迁移复杂度仍被轻描淡写

#### 结论

`V3.2` 已经提出了新的保存命令，这是正确的。

但从当前项目代码来看，现有保存逻辑仍明显围绕旧模型组织：

- 前端传 `transcripts[]`
- 后端走 `api_save_transcript`
- 数据结构默认只有单轨文本

因此这次迁移不只是“新增一个命令”，而是：

- 事件模型
- 前端缓存
- 持久化命令
- 旧数据兼容
- reload 恢复逻辑

的一次协同改造。

#### 建议

建议在 PRD 中增加一个简短的迁移策略说明：

1. 旧 `api_save_transcript` 是否保留兼容期
2. 新旧命令如何并存
3. IndexedDB 是否同步保存 `raw_text`
4. `getTranscriptHistory()` 是否要扩展为双轨数据
5. meeting details / pagination 是否只展示 normalized，还是支持查看 raw

这些不一定都要第一版实现，但必须在文档里明确“哪些要改、哪些暂不改”。

## 中优先级问题

### P2-1：snapshot hash 是进步，但还不是完整的“规则证据本体”

#### 结论

`terminology_snapshot_hash` 是这版的一个重要改进，值得保留。

但它的作用更像是：

- 审计索引
- 快照指纹

而不是完整的证据本体。

#### 原因

只保存 hash，未来可以知道：

- 规则集和现在不一样

但不能单独回答：

- 当时具体有哪些术语条目
- 哪条规则来自哪个术语包
- 哪次导入覆盖了什么

如果没有保留历史快照或导入批次历史，这个 hash 只能证明“曾经存在一组规则”，而不能完整重建那组规则。

#### 建议

建议 PRD 明确两层能力的区别：

- V1：保存 `terminology_snapshot_hash`
- Future：保存术语表历史快照或批次级快照

这样既不要求第一版过度实现，也不会让人误解“有 hash 就等于完整可回放”。

### P2-2：前端双轨扩展方案合理，但对恢复链路的影响还可以再写清楚

#### 结论

`TranscriptUpdate` 双轨扩展是 `V3.2` 的重要进步。

但当前前端不只存在“实时显示”这一条链路，还存在：

- `TranscriptContext` 缓冲
- IndexedDB 持久化
- reload 后 transcript history 恢复
- meeting details 的分页读取

如果 PRD 只强调实时监听和停止录音保存，而不明确这些衍生链路是否同步升级，研发很容易只改一半。

#### 建议

建议在文档中补一句明确说明：

- 双轨扩展不仅影响实时事件，还影响本地恢复与历史读取链路

最低限度建议列出需评估的受影响模块：

- `TranscriptContext`
- IndexedDB service
- `getTranscriptHistory`
- `useRecordingStop`
- `storageService`

## 项目管理与实施判断

### 当前判断

到了 `V3.2`，问题已经不再是“这个方案靠谱吗”，而是：

- “它是否已经精确到可以减少研发返工”

从这个标准看，`V3.2` 已非常接近可开发版本，但仍有几个地方建议再收口。

### 建议推进方式

我不建议重写 `V3.2`，而建议直接做一个很轻量的 `V3.2.1`，只修最后几处实现级问题：

1. 将 `L1 tokenizer` 的选型从“已定事实”改成“待验证方案”
2. 为 `L3` 队列增加明确的状态迁移和持久化说明
3. 明确“全文 N 处统一接受”属于简化策略而非天然安全策略
4. 把当前前端链路从 `page.tsx` 纠正为真实的上下文/服务层职责
5. 明确双轨扩展对恢复链路和历史读取链路的影响

### 是否可进入开发

我的判断是：

- 可以进入技术详细设计
- 可以开始最小 POC
- 可以拆接口任务
- 但不建议立刻开始全量编码实现

原因不是因为方向还不对，而是因为最后这几处问题如果现在不写清楚，后续会引发横跨前端状态、IPC 命令、DB 模型和恢复逻辑的返工。

## 建议的 V3.2.1 修订项

### 必改项

1. 将 `6.2` 中的 `cl100k_base` 表述改为“待验证 tokenizer 方案”
2. 在 `8.2` 中补充 L3 队列的状态迁移和持久化入队说明
3. 在 `8.6` 中补充“全文匹配统一应用”的限制说明和风险边界
4. 在 `9` 与 `10` 章中将前端职责落点改为 `TranscriptContext + useRecordingStop + storageService`
5. 增加一句说明双轨扩展会影响 IndexedDB / transcript history / reload sync

### 建议改项

1. 把 `snapshot hash` 与“完整术语历史快照”明确区分
2. 对 L3 建议中的高风险术语增加“默认要求逐条展开确认”的说明
3. 补充旧 `api_save_transcript` 与新命令之间的兼容期策略

## 最终结论

`V3.2` 已经不是“还有大方向错误”的文档了。

它已经具备以下特征：

- 架构意识明显增强
- 审计意识明显增强
- 工程边界比前几版清晰得多
- 已开始考虑现有代码结构的真实约束

这说明文档已经接近“可开发”状态。

但如果目标是：

- 尽量减少研发在实现中途返工
- 尽量减少前后端对 PRD 的不同理解

那么我仍建议在正式进入开发前补一个 `V3.2.1` 小修版。

结论可以概括为：

- `V3.2` 已经足够好，可以进入详细设计和 POC
- `V3.2` 还差最后一点实现级精度，补完后会更适合作为正式开发依据
