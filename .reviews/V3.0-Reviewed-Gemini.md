## 总体评价

这份 V3 版本的 PRD 相比 V2 有了**质的飞跃**。它不仅吸收了先前的审查意见，还在合规性（法务审查、建议模式）、系统健壮性（降级策略、原子缓存刷新、幂等迁移）以及项目管理（Phase 0 基线测量、MVP 拆分）上做了极其严密的补充。目前的版本已经具备了极高的工程落地指导价值。

但在以“严谨、完整、批判性”的标准进行最终审视时，仍能发现几个在深水区隐藏的**技术边缘 case、并发陷阱以及 UX/合规的博弈冲突**。以下是针对 V3 版本的深入 Review 意见：

---

## 一、 技术与架构的深层隐患 (Deep Technical Risks)

### 1. L3 本地大模型的并发灾难 (OOM Risk in Concurrency)
* **问题描述**：在 `7.5 调用链路` 中，设计为录音停止后执行 `tokio::spawn(async { llm_correction_task(meeting_id) })`。如果用户连续进行了两场短会议（例如录制 5 分钟 -> 停止 -> 立即开启下一场 5 分钟 -> 停止），系统将同时 Spawn 两个后台 LLM 任务。
* **致命后果**：本地运行 `qwen2.5:7b` 本身就需要占用 5-6GB 的显存/内存。同时并发请求 Ollama 会导致显存瞬间击穿（OOM），轻则 Ollama 服务崩溃，重则导致用户整个操作系统卡死。
* **修改建议**：必须引入**全局 LLM 任务队列（Task Queue）或信号量（Semaphore）**，严格限制本地 LLM 校正任务的并发数为 `1`。后台任务应排队执行，并在前端给予用户“排队中 (Queued)”的状态提示；同时需处理**App 关闭时的优雅退出（Graceful Shutdown）**，避免丢失正在生成的校正建议。

### 2. L1 `initial_prompt` 的截断策略缺失 (Truncation Strategy)
* **问题描述**：PRD 中提到“控制在 200 token 以内”，但在 5.6 节的代码中仅仅是 `terms.join(", ")`，并没有体现**如果超过了怎么裁剪**的逻辑。
* **业务影响**：如果一个企业配置了 80 个 High 优先级的日语术语，组合起来极大概率超过 Whisper 的 224 tokens 限制。如果不主动截断，`whisper-rs` 底层可能会报错或产生不可预期的行为。如果随意截断（比如按数组截断），可能会把最重要的核心术语丢弃。
* **修改建议**：需要明确截断策略。例如：在超出 token 限制时，优先保留最近更新的（`updated_at DESC`）术语，或者在 UI 上明确提示用户“高优先级术语已超载，前 50 个生效，后 30 个已被降级”。

### 3. L2 正则与 Aho-Corasick 的理论冲突
* **问题描述**：15 节风险应对中提到“超 500 条时评估 Aho-Corasick”。但 Aho-Corasick 是纯字面量（Literal）的多模式匹配树，**原生不支持 `\p{Han}` 或 `(?<!...)` 这样的正则前瞻/后顾断言**。
* **修改建议**：如果未来真的遇到性能瓶颈，不能简单地“替换为 Aho-Corasick”。必须在文档中明确：如果启用 AC 自动机，则只能处理 `whole_word = false` 的纯子串替换；对于需要边界判断的词，仍需走 Regex。这是一个混合匹配引擎的架构，会显著增加复杂度。当前保持 Cow 优化的 Regex 是正确的，但对 AC 的预期需要修正。

## 二、 用户体验与认知负荷 (UX & Cognitive Load)

### 1. L3 “逐条确认”带来的操作疲劳
* **问题描述**：为了规避合规风险，V3 将 L3 改为了“仅建议，需用户逐条确认”。在一场 1 小时的会议中，可能会产生 100+ 处术语纠正。如果让用户在 UI 上点击 100 次 `[接受]`，这在实际业务中是反人类的，用户最终会盲目点击 `[全部接受]`，从而让合规设计形同虚设。
* **修改建议**：前端 UI 需要支持**按术语聚类（Group by Term）**的确认模式。
  * *当前设计*：按时间线展示 100 个散落的句子 Diff。
  * *优化设计*：提示“本次会议中，将 `TDI` 修正为 `トルエンジイソシアネート` 共 15 处，[一键接受这 15 处]”。这不仅大幅降低了认知负荷，也让用户的审查更聚焦于术语本身。

## 三、 数据库与状态管理的边缘 Case

### 1. L3 校正状态与原文修改的竞态条件 (Race Condition)
* **问题描述**：在 L3 异步生成校正建议（需要 10-30 秒）的过程中，如果用户在前端**手动修改了那段原文的文字**（例如修正了某个标点），当 L3 的结果返回并存入 `transcript_corrections` 时，它的 `original_text` 已经与数据库中真实的原文不匹配了。
* **修改建议**：在执行 `accept_correction` 替换文本时，需要进行**版本比对或严格的偏移量（Offset）/原文字符串匹配检查**。如果发现原文已经被用户手动修改，该条 L3 建议应当自动失效（标记为 `obsolete` 或 `conflict`），以防止 LLM 的旧建议覆盖了用户的手动精准修改。

### 2. CSV 导入的“静默覆盖”风险
* **问题描述**：11.2 节提到了 CSV 导入。如果用户导入的 CSV 中，`original` 字段与现有数据库中的术语冲突（比如同音字被指向了不同的替换词），系统是报错、跳过还是覆盖？
* **修改建议**：需要在 PRD 中明确 UPSERT（更新插入）的冲突解决策略。通常 B2B 系统的最佳实践是：基于 `original` 和 `language` 作为联合唯一键，如果冲突则默认覆盖（或在前端导入预览时给出“将覆盖 X 条现有规则”的警示）。
