# PRD_TERMINOLOGY_CUSTOMIZATION_V3.3 审查意见

## 审查范围

- 审查对象：`PRD_TERMINOLOGY_CUSTOMIZATION_V3.3.md`
- 参考代码库：`meetily-forked`
- 审查方式：基于当前代码架构、前端状态流、IndexedDB 恢复链路、Rust 数据模型和已有任务状态模式进行静态审查
- 审查目标：确认 `V3.3` 是否已经达到可交付研发进入详细设计和 POC 的程度，并识别仍可能造成实现歧义或返工的关键问题

## 总体结论

`V3.3` 是目前为止最成熟、最接近可开发版本的一版 PRD。

与前几版相比，这一版已经明显具备以下特征：

- 对现有项目真实代码结构的理解更准确
- 对审计、追溯、恢复链路、兼容迁移的考虑更完整
- 对 L3 风险边界的表达更克制、更工程化
- 对 Phase 0 验证、降级路径和前端状态改造有更清晰的交代

因此，我认可文档中“已具备进入技术详细设计和 POC 开发条件”的判断。

但是，`V3.3` 仍然没有完全达到“可直接交给研发无歧义开工”的程度。

本轮最核心的问题不是方向错误，也不是能力边界不清，而是一个数据建模层面的冲突：

- 文档把 **L3 任务队列状态**
- 和 **L3 建议审核状态**

混进了同一张表、同一个 `status` 字段。

这是当前版本中最需要优先修掉的问题。

如果这个问题不拆开，后续会在以下环节同时引发语义混乱：

- SQL 查询
- 状态迁移
- 前端展示
- 超时重试
- 审计逻辑
- 用户接受/拒绝建议

因此我的结论是：

- `V3.3` 已足够进入详细设计和 POC
- 但在正式全面开工前，仍建议先做一个小修版 `V3.3.1`
- `V3.3.1` 的首要目标应是拆开 L3 任务状态模型和建议审核状态模型

## 本轮结论摘要

### 明显进步

- `L1 tokenizer` 已从“既定事实”改成“待验证方案”
- `L3` 全文同串匹配被明确标记为“受控简化策略”
- 当前前端职责不再模糊写成 `page.tsx`，而是落到实际模块
- 迁移策略和受影响模块终于被单独成章
- 双轨数据流与恢复链路的影响范围被正面承认

### 仍然不过线的部分

- `L3` 队列状态与建议审核状态混用同一 `status`
- 前端双轨扩展虽然写得更准，但实际改造量仍应进一步高亮
- `L3` 对短文本/高频短子串的控制还可更严格
- `meeting details` 中 raw 查看入口的产品边界仍未完全定义

## 严重问题

### P0-1：L3 任务状态与建议审核状态被混进同一张表、同一 status 字段

#### 结论

这是 `V3.3` 当前最重要、最应优先修正的问题。

#### 当前文档中的冲突

一方面，文档在审计和建议层面明确把 `status` 当作审核状态使用，例如：

- `accepted`
- `reviewed_by`
- `reviewed_at`

另一方面，在 `L3 持久化任务队列` 和数据库设计里，又把同一个 `status` 扩展成：

- `queued`
- `running`
- `done`
- `failed`
- `timeout`

这意味着当前设计把两类完全不同的对象混在了一起：

1. **L3 任务**
2. **L3 建议**

#### 为什么这是结构性问题

这两类状态不是一个维度：

- 任务状态回答的是“LLM 任务执行到了哪一步”
- 建议状态回答的是“用户是否接受了这条建议”

它们的生命周期也完全不同：

- 任务通常一场会议一次或少量几次
- 建议通常是一场会议多条，且每条都可独立处理

如果用同一张表、同一字段承载，会出现以下问题：

- 一条记录到底是“任务元数据”还是“具体建议”语义不清
- `queued` 与 `accepted` 无法在一个统一状态机里表达
- SQL 查询需要不断区分“meta 记录”和“真实建议”
- 前端拿到 `status` 后很难判断该显示队列状态还是审核状态
- 重试逻辑、统计逻辑、审计逻辑都会变复杂

#### 建议

必须拆表或至少拆状态模型。

最推荐的方式是新增一张独立任务表，例如：

- `l3_correction_jobs`

承载：

- `meeting_id`
- `status`：`queued | running | done | failed | timeout`
- `error_detail`
- `attempt_count`
- `created_at`
- `updated_at`

而原有的：

- `transcript_corrections`

只保留建议本身及其审核状态，例如：

- `pending`
- `accepted`
- `rejected`
- `obsolete`

这是当前版本最值得优先处理的修订项。

## 高优先级问题

### P1-1：双轨扩展已经写得更准确，但对现有恢复链路的改造成本仍应进一步强调

#### 结论

`V3.3` 已经明确指出双轨扩展会影响：

- `TranscriptContext`
- `indexedDBService`
- `useTranscriptRecovery`
- `storageService`
- `useRecordingStop`

这是非常大的进步。

但从当前代码现状看，这仍然是一项跨多个前端子系统的改造，而不只是“加两个字段”。

#### 当前代码现实

现有系统中：

- `TranscriptContext` 仍以单轨 `text` 为主组织转录状态
- IndexedDB 当前也是把 transcript update 原样落盘
- 恢复流程仍按单轨 transcript 重建

这意味着双轨扩展会同时影响：

- 运行中实时缓冲
- 本地容灾缓存
- 重启恢复
- 停止录音保存
- 历史 transcript 再入库

#### 建议

建议在 PRD 的工作量估算或实施策略中再明确一句：

- 双轨改造是一个“恢复链路级别”的改造包

否则很容易在排期时被低估成前端小改。

### P1-2：L3 全文同串匹配的风险说明已明显改进，但对高频短片段仍可更严格

#### 结论

`V3.3` 已经正确地把“全文 N 处统一接受”标记为受控简化，而非天然安全策略，这是非常正确的收口。

#### 剩余问题

当前文档主要通过以下方式约束风险：

- LLM 尽量返回较长 `original_span`
- `<3` 字符时执行额外边界检查
- 高风险代码类术语默认逐条确认

这些方向都对，但在中文/日文场景中，3-4 个字符仍然可能是高频共享片段。

例如：

- 常见化学后缀
- 高频缩写前缀
- 共享部件词

即使达到 3-4 个字符，也仍不一定适合“全文统一接受”。

#### 建议

建议再补一条更强的限制规则：

- 当 `original_span` 命中次数异常高、或疑似常见词缀片段时，默认禁用“接受全部”，仅允许逐条展开确认

这样可以进一步降低全文泛化带来的误操作风险。

### P1-3：fancy-regex 的降级思路正确，但验收标准还可以更明确地与降级路径联动

#### 结论

文档已将 `fancy-regex` 明确定义为需要在 Phase 0 验证的关键依赖，这一点是合理的。

但当前文档更多强调“如果失败就降级为子串匹配”，还可以再进一步写清楚：

- 一旦降级，哪些验收标准同步放宽
- 哪些语言能力需要在 UI 或文档中明确标注为已知限制

#### 建议

建议在风险与应对、验收标准或 MVP 边界中补一句：

- 如果 `fancy-regex` 验证失败，则日/中 whole-word 能力不纳入 Phase 1A 的验收承诺

这样能避免“实现已降级，但验收仍按高标准要求”的错位。

## 中优先级问题

### P2-1：前端 useRef 双轨缓冲优化是加分项，但代码落点还可更精确

#### 结论

`V3.3` 已经意识到长会议场景下 React state 不可变数组追加带来的性能与 GC 压力，这是非常好的工程意识。

#### 问题

当前示例写在事件监听建议模式附近，但如果要更贴近现有代码结构，最好明确这段逻辑应落在：

- `TranscriptContext`

而不是服务层。

#### 建议

建议在文档中把 `useRef` 双轨 buffer 的落点明确标为：

- `TranscriptContext.tsx`

这样更方便研发直接定位改造位置。

### P2-2：meeting details 中 raw 查看入口的产品边界仍需再明确

#### 结论

PRD 已经把 `meeting details` 纳入受影响模块，这很好。

但当前仍未明确：

- `raw_transcript` 在 V1 是否默认对普通用户可见

这不是纯技术问题，而是产品与审计边界问题。

#### 建议

建议在文档中明确以下其中一种策略：

1. `raw_transcript` 仅供审计模式/管理员模式查看
2. `raw_transcript` 默认隐藏，仅在调试或合规审查入口中展开
3. `raw_transcript` 普通用户可见，但有明显标签区分

如果这一点不定，前端实现会在可见性、权限和交互上反复摇摆。

## 项目管理与实施判断

### 当前判断

到了 `V3.3`，PRD 的主问题已经不是“有没有想明白”，而是：

- “还差哪几个点，能让研发更少返工”

从这个角度看，`V3.3` 已经非常接近正式开发依据。

### 建议推进方式

不建议重写 `V3.3`，而建议只做一个极轻量的 `V3.3.1`：

1. 拆开 L3 任务状态和建议审核状态
2. 明确 raw 查看入口的产品边界
3. 强化双轨恢复链路的工作量提示
4. 对高频短子串的 L3 全文匹配再加一道限制

这几项都属于小范围结构修补，不会推翻现有方案。

### 是否可进入开发

我的判断是：

- 可以进入技术详细设计
- 可以开始 POC
- 可以拆分接口与数据层任务
- 但正式全面开工前，仍建议先做一次 `V3.3.1` 小修

原因非常简单：

- 现在剩下的问题数量不多
- 但其中有一个是数据模型级冲突
- 这种问题如果不先修，后面返工代价会高于现在改 PRD 的代价

## 建议的 V3.3.1 修订项

### 必改项

1. 新增 `l3_correction_jobs` 表，单独承载任务队列状态
2. 保持 `transcript_corrections.status` 仅表达建议审核状态
3. 在 `meeting details` 相关章节明确 raw 查看入口是否默认隐藏
4. 在 L3 风险章节中补充“高频短片段默认禁用接受全部”

### 建议改项

1. 在 `10.4` 节将 `useRef` 双轨 buffer 的实现落点明确到 `TranscriptContext.tsx`
2. 在实施计划中把“双轨恢复链路改造”显式标记为前端高复杂度子任务
3. 在验收标准中补充降级后验收边界的说明（针对 `fancy-regex` 失败场景）

## 最终结论

`V3.3` 已经达到一个很重要的节点：

- 不再需要大幅重写
- 不再需要重新定义主问题
- 不再需要重新讨论能力边界

它已经具备进入详细设计和 POC 的条件。

但如果目标是：

- 把研发歧义进一步压低
- 把实现中的结构返工风险压低

那么我仍建议先补一个小修版。

结论可以概括为：

- `V3.3` 已经足够成熟，可以进入详细设计与 POC
- `V3.3` 还差最后一个关键建模修正，修完后才更适合作为正式实施依据
