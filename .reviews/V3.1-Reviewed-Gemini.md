# PRD_TERMINOLOGY_CUSTOMIZATION_V3.1.md 深度 Review 报告

> **Reviewer**: Gemini-3.1-Pro-Preview
> **Date**: 2026-04-27
> **Target File**: `PRD_TERMINOLOGY_CUSTOMIZATION_V3.1.md`

## 总体评价

这是一份非常出色的 PRD。V3.1 版本不仅吸收了前期多轮审查的建议，还将架构的严密性推向了企业级标准。尤其是**“L0 原始证据链强制保留”、“L3 细粒度补丁模型”以及“全局 Semaphore 串行队列”**的引入，彻底解决了合规审计和硬件资源 OOM 的痛点。

然而，在极致的边缘 Case（Edge Cases）和正则表达式的底层执行机制上，V3.1 仍存在几个可能导致功能部分失效的**深水区逻辑陷阱**。以下是针对 V3.1 版本的最后一轮深度 Review：

---

## 一、 致命逻辑漏洞：L2 捕获组边界的“吞噬”效应 (The "Consuming" Effect)

**问题描述**：
在 7.2 节中，为了绕过 Rust 标准 `regex` 库不支持前后瞻（Look-around）的限制，PRD 提出了使用捕获组的替代方案：`(^|[^\p{Han}...])term($|[^\p{Han}...])`。
PRD 中提到：“中间的边界字符被第一个匹配消耗，第二个术语可能不匹配——但由于术语间通常有标点或空格分隔，实际影响有限”。**这是一个严重的误判。**

**原理剖析**：
假设文本为 `ポリウレタン、過酸化物`（两个术语被一个逗号分隔）。
1. 正则引擎执行匹配，命中 `ポリウレタン`，并将前面的行首（`^`）和后面的逗号（`、`）作为组 1 和组 2 **消耗（Consume）**掉。
2. 此时，正则引擎的匹配指针已经移动到了逗号**之后**（即 `過` 字前面）。
3. 引擎继续向后搜索 `過酸化物`。但是，`過酸化物` 的正则要求其前面必须是“行首或非日文字符”。此时指针前面的字符是逗号，但逗号已经被上一次匹配“吃掉”了，引擎无法回溯去看那个逗号。因此，`過酸化物` **将匹配失败**。

**业务影响**：
只要两个术语之间只有一个标点符号或空格（这是最常见的自然语言书写方式），**紧跟在后面的所有术语都会失去 L2 校正保护**。

**修正建议**：
1. **方案 A（推荐）**：果断引入 `fancy-regex` crate。它在底层使用标准 `regex` 的 DFA 引擎保证性能，但在遇到前后瞻断言 `(?<=...)` 和 `(?=...)` 时会平滑回退到回溯引擎。对于术语匹配，这点性能损耗远小于“吞噬”效应带来的准确率灾难。
2. **方案 B（多轮重放）**：如果坚持使用标准 `regex` 的捕获组，必须修改 7.3 节的 `apply_terminology_correction` 函数，在一个 `while` 循环中不断执行 `replace_all`，直到文本不再发生任何变化为止（但这会带来死循环风险和额外的性能开销）。

## 二、 鲁棒性隐患：L3 偏移量对齐与多频词定位 (Offset Alignment Risk)

**问题描述**：
在 8.3 节（建议数据结构）和 10.2 节中，系统需要记录 `char_offset`。但根据 8.6 节的 LLM Prompt，LLM 仅返回 `{"original": "誤った文字列", "suggested": "修正後の文字列"}`，**LLM 并不会返回偏移量**（LLM 也不擅长数字符）。

这意味着，Rust 后端在接收到 LLM 的 JSON 后，必须自己在 `normalized_transcript` 中通过 `str::find(&original_span)` 去计算 `char_offset`。
如果一个错误的词在同一段话中**出现了多次**（例如：“这个 *H二二五* 比那个 *H二二五* 更危险”），`str::find` 只能找到第一个匹配项的偏移量。如果 LLM 实际上只打算修复第二个（或者对两个给出了不同的修复理由），这种基于字面量反查偏移量的逻辑就会发生错位，导致“指鹿为马”的 UI 高亮错误。

**修正建议**：
* **前端对齐策略**：废弃严格的绝对 `char_offset` 依赖。将 L3 的建议视作基于上下文的**“查找与替换（Find & Replace）”规则**。在 UI 展示时，前端在当前 chunk 内全局高亮所有匹配的 `original_span`，用户点击接受时，将该 chunk 内的所有该错误词一并替换。
* 或者在 LLM Prompt 中要求输出上下文片段（如 `{"context": "比那个H二二五更", "original": "H二二五"}`），后端通过上下文来精确定位。

## 三、 工程优化：L1 Token 截断的精确化 (Precise Token Truncation)

**问题描述**：
在 6.2 节的 Token 截断策略中，PRD 建议使用估算值（“日语约 1.5 token/词，中文约 2 token/词”）。
由于 Whisper 的 Tokenizer（如 `cl100k_base` 或 `p50k_base`）对低频化学词汇的切分极不规律，长化学名可能被切碎成十几个 token，这种粗略的乘法估算极易导致**预算超载报错**或**空间浪费**。

**修正建议**：
在 Rust 后端引入 `tiktoken-rs` 或 `tokenizers` crate。在 `build_initial_prompts` 缓存构建阶段，直接使用 Whisper 的 BPE 词表对术语进行**真实的 Token 计数（Exact Count）**。这不仅能 100% 避免超出 224 tokens 的硬限制，计算开销在启动时也仅为几毫秒，完全可忽略不计。

---

## 四、 状态流转盲区：App 退出与 L3 任务恢复 (Lifecycle Management)

**问题描述**：
在 8.2 节提到“支持优雅退出：监听 AppHandle 的 shutdown 信号，提前中断”。
但录音结束触发 L3 建议生成（可能需要 10-60 秒）。如果用户在点击“停止录音”后，习惯性地直接关闭了 Meetily 软件，当前正在排队的 L3 任务就会被丢弃。当用户下次打开软件查看该会议记录时，会发现 L3 建议永远停留在 `pending` 或根本没有生成。

**修正建议**：
1. **重启恢复机制**：在应用启动初始化阶段（`setup.rs`），查询 `transcript_corrections` 或 `meetings` 表中是否有“已转录完毕但未完成 L3 校正”的记录，如果有，自动将其重新推入全局 Semaphore 队列中恢复生成。
2. **UI 退出阻断（可选）**：如果后台有正在运行的 L3 任务，用户点击关闭窗口时，弹出 Toast 提示：“深度校正正在后台生成，软件将最小化到托盘继续处理”，而非直接 Kill 进程。

---

## 五、 总结与定调

V3.1 版本是一份**准度极高、可以直接交付研发**的架构级 PRD。它在业务价值和工程落地之间找到了极佳的平衡点。

上述提出的四个问题属于**技术详细设计（Detailed Design）阶段的防御性编程**范畴。不需要再发布 V3.2，建议直接将本 Review 报告作为**“开发避坑指南（Development Caveats）”**附在开发任务的 Epic 之下，在研发工程师编写 Rust 正则解析和状态管理代码时作为强制约束即可。
